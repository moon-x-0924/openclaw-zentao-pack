# 后台变更通知覆盖清单

这份清单用于统一查看两条链路的通知覆盖情况：

- 禅道后台人工操作后的即时通知
- 企微会话触发技能包脚本后的即时通知

目标不是列全量禅道动作，而是优先盯住“改完后业务会期待马上通知到人”的高时效场景。

当前工作原则：

- 后台操作优先。凡是禅道后台人工操作后业务上应该通知的场景，都必须纳入通知覆盖范围，不能只保证脚本入口能通知。
- 脚本链路要与后台语义对齐，但不能替代后台覆盖。

当前与服务器热修相关的仓库落点：

- 热修源码副本：`references/server-hotfixes/task-model-2026-04-16.php`
- 回写说明：`docs/overview/task后台通知热修回写说明.md`

## 已确认对齐的场景

| 对象 | 场景 | 禅道后台人工操作 | 技能包脚本操作 | 说明 |
| --- | --- | --- | --- | --- |
| task | 转派 | 已通过服务器热修补齐 | 暂无独立 `assign-task` 脚本 | 后台链路依赖 task model 兜底直推 bridge |
| task | 完成并转派 | 已通过服务器热修补齐 | `update-task-status` 当前不带 `assigned-to`，无同类入口 | 后台会补发 `finished` + `assigned` |
| task | 激活 / 开始 / 重启 / 暂停 / 取消 / 关闭 | 已通过服务器热修补齐 webhook 投递 | `update-task-status` 可覆盖部分状态 | 当前链路重点风险已从“事件发不出”转为“规则是否命中” |
| bug | 单独转派 | 原有链路可用 | `assign-bug` 已按前后值自动补发 `assignee_changed` | 脚本本轮已同步到“按实际结果补通知” |
| bug | 状态流转且改负责人 | 依赖原生 action / webhook | `update-bug-status` 已按前后值同时判断 `status_changed` / `assignee_changed` | 避免脚本只发状态通知 |
| story | 状态流转且改负责人 | 依赖原生 action / webhook | `update-story-status` 已按前后值同时判断 `status_changed` / `assignee_changed` | 本轮已补齐脚本侧双通知语义 |
| story | 评审并改负责人 | 待专项核查 | `review-story` 已按前后值自动补发通知 | 先保证企微会话脚本不丢通知 |

## 当前已知遗漏

| 对象 | 场景 | 当前状态 | 根因判断 | 建议优先级 |
| --- | --- | --- | --- | --- |
| task | `confirmed / started / activated / restarted` | webhook server 接得住动作，但大概率命不中现有规则 | 规则文件只有 `done / blocked / closed` 一类 task 状态通知 | 高 |
| bug | `confirmed / bugconfirmed` | webhook server 接得住动作，但当前无明确通知规则 | bridge 已支持动作识别，规则层未补语义 | 中 |
| story | `changed / reviewed` 引起的字段变更通知 | 文档里有 `field_changed / priority_changed` 规则，代码侧尚未产出对应事件 | `wecom_notify` 当前只正式支持 `status_changed / assignee_changed` | 高 |
| story / bug / task | `created` 新建通知 | 规则与模板已写，脚本与 webhook 侧都还没正式发 `created` 事件 | 事件模型未补到执行层 | 高 |
| release / testtask 等对象 | 后台变更即时通知 | 目前没有统一通知模型 | 现阶段 MVP 还聚焦 story / bug / task | 暂缓 |

## 这轮已落地的技能包同步规则

- 所有会修改 `story / bug / task` 状态或负责人的脚本，先抓变更前快照，再按变更后实际值决定是否发通知。
- 如果一次操作同时改了状态和负责人，脚本必须允许同时补发 `status_changed` 与 `assignee_changed`。
- 通知失败视为附加动作失败，不回滚禅道业务主操作。
- 输出结果同时保留 `notification` 摘要和 `notifications` 明细，方便企微回执和人工排查。

## 下一轮建议排查顺序

1. task 后台高时效动作全量核查：`assign / finish+assign / pause / activate / restart / close / cancel`
2. story 后台动作核查：重点看 `reviewed / changed / close / activate`
3. 把 `created` 事件真正接入脚本侧与 webhook 侧，而不是只停留在规则文档
4. 再决定是否把 release / testtask 纳入同一套通知语义

## 排查时固定检查项

1. 对象数据是否真的改了：`zt_task / zt_bug / zt_story`
2. 动作记录是否真的落了：`zt_action`
3. bridge 是否收到 webhook：`journalctl -u zentao-wecom-webhook.service`
4. 企微是否真正发出：看 `notification-audit` 与企微返回码
