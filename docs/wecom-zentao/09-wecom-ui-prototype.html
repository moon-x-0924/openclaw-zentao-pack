发布上线
  • 对应 SOP Step 21-22：创建发布、上线执行。
  • 重点是上线检查卡和发布确认卡。
  • 用于评审“发布负责人是否能在企微完成确认”。
验收关闭
  • 对应 SOP Step 23-24：上线后业务验收、关闭需求与事项。
  • 重点是验收概览和关闭事项确认。
  • 用于评审“整个流程是否能形成最终闭环”。
产品角色工作流
  • 围绕产品建立、需求提出、评审结论、上线验收展开。
  • 重点关注“我要提什么”“评审结论是什么”“上线后是否能验收通过”。
  • 机器人适合做需求草稿、评审摘要、验收概览。
产品 / 提需求
  • 聚焦需求提报、字段补齐、评审前信息整理。
  • 最适合用机器人先生成需求草稿，再进评审。
产品 / 做验收
  • 聚焦业务目标、核心流程、权限、页面/接口验收。
  • 最适合用摘要卡快速判断是否进入关闭阶段。
研发角色工作流
  • 围绕我的任务、状态流转、工时、自测、Bug 修复展开。
  • 重点关注“今天干什么”“任务能否结束”“还有哪些 Bug 待修复”。
  • 机器人适合承接高频动作，自建应用承接复杂对象详情。
研发 / 做任务
  • 聚焦开始任务、记录工时、完成任务。
  • 强调高频状态流转和确认卡。
研发 / 修 Bug
  • 聚焦“指派给我”的 Bug、修复回填、待验证状态。
  • 强调从 Bug 卡直接进入修复闭环。
测试角色工作流
  • 围绕测试范围、执行用例、提 Bug、回归、准出展开。
  • 重点关注“当前阻塞项是什么”“是否允许上线”。
  • 机器人适合输出摘要与阻塞项，自建应用承接复杂 Bug 补录。
测试 / 执行测试
  • 聚焦测试单、用例执行进度和失败项处理。
  • 强调测试执行卡和失败直达提 Bug。
测试 / 准出判断
  • 聚焦阻断 Bug、回归完成度、发布说明是否齐全。
  • 强调准出卡是管理决策输入。
管理角色工作流
  • 围绕项目准备度、发布准备度、验收与关闭展开。
  • 重点关注“是否能进入下一阶段”“风险在哪”。
  • 机器人更像驾驶舱，自建应用更像结构化管理台。
管理 / 项目推进
  • 聚焦项目准备度、任务完成率、风险项。
  • 强调项目推进驾驶舱价值。
管理 / 上线关闭
  • 聚焦上线检查、发布确认、关闭事项。
  • 强调最后一公里的管理收口能力。
跨角色协作总览
  • 把产品、管理、研发、测试、发布、验收串成同一条协作链。
  • 重点是“谁把什么交给谁”，而不是单角色局部效率。
  • 机器人价值体现在上下文承接和跨角色信息收口。
跨角色 / 角色接力
  • 展示需求草稿、任务、提测、准出、发布结果如何逐步接力。
  • 适合评审“机器人如何承接角色之间的信息传递”。
跨角色 / 上线闭环
  • 展示测试准出、发布上线、业务验收、关闭事项的闭环关系。
  • 适合评审“最终为什么不是发完版就结束”。

当前场景注意点

系统准备 · 注意点
  • 第一屏先暴露身份映射、权限缺口和同步结果，不要让用户先试错。
  • 优先展示“谁没绑定”“哪些角色缺权限”“下一步去哪里修”。
  • 适合产出权限检查卡、身份同步卡、失败恢复卡。
成品建议：自建应用承接成员映射表、角色权限矩阵和同步日志页。
产品建立 · 注意点
  • 重点不是直接创建成功，而是先把产品名、模块、归属和负责人草案收拢清楚。
  • 机器人应优先展示产品初始化摘要和待确认模块，不要一上来就长表单。
  • 适合产出产品草稿卡、模块确认卡、重名冲突提示卡。
成品建议：右侧成品应是产品初始化确认页，支持产品名、模块树、负责人和产品线编辑。
需求发起 · 注意点
  • 第一版只保留最小需求创建，不做正式评审和复杂拆解。
  • 必须突出需求标题、负责人、优先级和是否进入开发。
  • 适合产出需求创建结果卡、进入开发确认卡、需求详情入口。
成品建议：应用页优先做需求详情页，只承接补充描述、负责人和优先级。
项目迭代 · 注意点
  • 关注项目准备度、团队是否齐、执行时间是否冲突，而不是只展示创建成功。
  • 卡片要能快速说明哪些需求已挂接、哪些执行还缺团队或负责人。
  • 适合产出项目准备清单卡、执行缺口提示卡。
成品建议：应用页重点做执行规划台，承接团队、周期、需求挂接和缺口修正。
研发执行 · 注意点
  • 高频动作必须短链路，优先支持我的任务、开始任务、完成任务、发起提测。
  • 卡片里必须显示任务 ID、当前状态和直接动作，不要求第一版就做完整工时体系。
  • 适合产出我的任务卡、开始确认卡、完成确认卡、提测确认卡。
成品建议：应用页先做任务详情和简单备注补录，不把复杂说明塞进聊天卡片。
测试结果 · 注意点
  • 第一版只保留“通过”或“退回研发”两种测试结论，不做完整准出体系。
  • 问题描述尽量简短，复杂复现信息再跳到应用页补录。
  • 适合产出提测确认卡、测试结果卡、问题详情入口。
成品建议：应用页承接测试结果详情和最小 Bug 挂载，机器人负责结论与推进。
发布上线 · 注意点
  • 所有上线动作都应强调风险、回滚方案、影响范围和确认痕迹。
  • 发布负责人最关心的是“现在能不能发”和“缺什么不能发”。
  • 适合产出上线检查卡、发布确认卡、失败恢复卡。
成品建议：应用页应展示发布详情、回滚方案、变更清单和上线记录。
关闭确认 · 注意点
  • 第一版只判断能不能关闭，不做复杂验收清单和正式归档流。
  • 必须把测试结果、任务完成情况和遗留说明汇总到一张卡里。
  • 适合产出关闭前摘要卡、关闭确认卡。
成品建议：应用页应展示关闭确认页，支持查看需求详情、测试结果和遗留说明。
产品经理 · 注意点
  • 第一版只服务产品经理的三件事：创建需求、看状态、确认关闭。
  • 不要让产品经理在企微里处理复杂 PRD 拆解、评审编排和迭代规划。
  • 适合产出需求创建卡、状态推进卡、关闭确认卡。
成品建议：重点页面是需求创建页、需求详情页和关闭确认页。
产品 / 提需求 · 注意点
  • 核心是用最少字段快速创建需求，不要把 PRD 拆解做成第一版前提。
  • 必须给产品经理可编辑的标题、描述、负责人、优先级和产品/模块字段。
成品建议：右侧优先展示需求创建页和需求详情页，而不是复杂草案工作台。
产品 / 确认关闭 · 注意点
  • 第一版重点是判断是否满足关闭条件，而不是做完整业务验收。
  • 关闭动作前必须让产品经理看到测试结果、任务完成情况和遗留说明。
成品建议：展示关闭前摘要页和关闭确认卡，遗留说明作为补充信息查看。
研发 · 注意点
  • 研发最关心今天做什么、任务能否结束、还有哪些 Bug 等我修。
  • 机器人必须保证高频动作一屏闭环，复杂详情再跳应用。
成品建议：工作台默认落在“我的任务 + 我的 Bug + 待确认动作”。
研发 / 做任务 · 注意点
  • 强调开始任务、完成任务、发起提测三个关键动作。
  • 任务详情和补充说明更适合放应用页，聊天里只保留推进动作。
成品建议：展示任务详情页、完成确认卡和提测确认卡的组合效果。
研发 / 修 Bug · 注意点
  • 先给“指派给我”的 Bug 摘要，再进入修复回填。
  • 解决方案、附件、验证说明应放详情页而不是消息流。
成品建议：展示我的 Bug 列表和 Bug 回填页。
测试 · 注意点
  • 测试第一关注是范围、阻塞项和能否准出，不是对象创建本身。
  • 机器人适合给结论，自建应用适合给完整证据和附件。
成品建议：工作台默认突出准出状态、阻塞 Bug 和待回归项。
测试 / 执行测试 · 注意点
  • 第一版重点是输出测试结论，不必展开完整测试流程和报告结构。
  • 复杂复现步骤、环境信息、附件都交给应用页，聊天里保留通过或退回。
成品建议：展示测试结果页和问题详情页的联动。
测试 / 准出判断 · 注意点
  • 准出卡只说最关键结论：能否上线、阻断项、待回归项、发布说明缺口。
  • 适合继续查看，不适合第一屏塞完整测试报告。
成品建议：展示准出摘要卡和完整报告入口。
管理 · 注意点
  • 管理最关心的是“当前能否进入下一阶段”和“风险在哪”。
  • 右侧应更像驾驶舱，而不是单对象表单。
成品建议：工作台聚焦准备度、完成率、阻塞项和上线条件。
管理 / 项目推进 · 注意点
  • 突出准备度、逾期任务、团队缺口和阶段风险。
  • 适合做项目推进驾驶舱，不适合做复杂录入页。
成品建议：展示项目推进看板和阻塞项明细入口。
管理 / 上线关闭 · 注意点
  • 核心是“是否可发”“发后怎么回滚”“关哪些对象”。
  • 确认卡必须足够强，避免误点带来流程事故。
成品建议：展示上线检查页、发布确认页和关闭项清单。
跨角色协作 · 注意点
  • 第一版只需要讲清一条最短闭环：需求创建、任务执行、测试结果、确认关闭。
  • 右侧应突出当前卡点和下一责任人，而不是完整泳道和复杂交接物。
成品建议：展示最短闭环总览卡，帮助团队看清一条需求如何被推进到关闭。
跨角色 / 角色接力 · 注意点
  • 每一段都要写清“谁输出什么给下一个角色”。
  • 适合做交接物清单，不适合做字段表单。
成品建议：展示交接链路板：需求草案 -> 任务 -> 提测 -> 准出 -> 发布。
跨角色 / 上线闭环 · 注意点
  • 必须强调验收通过、关闭事项和遗留问题留档三件事一起成立才算闭环。
  • 适合把测试、发布、产品和业务确认串成最终回执。
成品建议：展示上线闭环板和关闭项确认页。

通用卡片设计基线

结果卡
  • 第一屏先给结论，不让用户再读一遍解释。
  • 必须显示对象 ID、状态、归属上下文。
  • 底部最多保留 2~3 个主动作，避免卡片膨胀。
确认卡
  • 当前状态、目标状态、风险影响要并列展示。
  • 确认按钮必须显著高于取消按钮。
  • 卡片里直接提示有效时间和审计记录。
候选选择卡
  • 不直接猜测同名对象，统一走候选卡。
  • 用编号 + 产品/执行上下文帮助快速区分。
  • 移动端优先数字化选择，降低输入成本。
失败卡
  • 不要只说失败，要给出具体原因和恢复动作。
  • 优先提供“打开应用”“查看帮助”等下一步。
  • 权限问题和系统问题在视觉和文案上要区分开。
