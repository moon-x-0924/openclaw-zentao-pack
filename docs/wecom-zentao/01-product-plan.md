# 企微对话式禅道助手产品方案

## 1. 产品定位

- 以 `openclaw-zentao-pack` 作为现有业务内核，补齐产品边界、交互约束、权限控制和实施路径。
- 让用户在企业微信里完成禅道高频操作，把复杂操作分流到自建应用承接。

## 2. 角色定义

| 角色 | 核心目标 | 高频动作 |
| --- | --- | --- |
| 产品 | 创建产品、整理需求、发起评审 | 创建产品、创建需求、评审需求 |
| 研发 | 查看和推进任务、处理 Bug | 我的任务、开始任务、完成任务、处理 Bug |
| 测试 | 看测试单、提 Bug、查准出 | 创建 Bug、查测试单、测试准出 |
| 项目经理 / 交付 | 看执行和发布状态 | 查执行、查发布、验收概览 |
| 负责人 | 审批高风险操作 | 发布确认、关闭确认、状态流转确认 |
| 管理员 | 配置路由、身份、权限、审计 | 绑定用户、排错、查看审计 |

## 3. 业务能力地图

| 能力域 | 核心内容 | 优先级 |
| --- | --- | --- |
| 身份与会话 | 企微身份识别、禅道映射、上下文 | P0 |
| 高频查询 | 我的任务、我的 Bug、执行、测试单、准出、上线检查 | P0 |
| 核心创建 | 产品、需求、任务、Bug、测试单、发布 | P0 |
| 核心流转 | 任务、需求、Bug、发布、测试单状态更新 | P0 |
| 风险控制 | 二次确认、权限校验、审计追踪 | P0 |
| 管理视图 | 验收概览、关闭准备度、关闭阻塞项 | P1 |
| 应用承接 | 自建应用复杂表单和详情页 | P1 |

## 4. MVP 边界

- 优先完成企微机器人接入。
- 打通身份映射、回调识别、查询、创建、流转。
- 建立危险操作确认与审计闭环。
- 自建应用首期只承接复杂表单和草案确认，不做完整工作台。

## 5. 分阶段路线图

- Phase 1：身份映射、回调解析、统一路由、核心查询与执行。
- Phase 2：多轮补问、候选选择、管理视图、失败恢复优化。
- Phase 3：自建应用工作台、复杂表单、主动提醒、历史与审计面板。

## 6. 高频场景

- `我的任务`
- `查执行 4`
- `测试准出 4`
- `在执行 4 创建任务，标题是补齐企微映射`
- `完成任务 33`
- `提个 Bug`

## 7. 用户故事

- 作为研发，我希望在企业微信里直接看我的任务，这样不用频繁切回禅道后台。
- 作为测试，我希望快速看测试单和准出状态，这样能尽快判断是否满足上线条件。
- 作为产品，我希望通过对话快速创建需求和发起评审，这样能缩短沟通链路。
- 作为项目经理，我希望在聊天里直接获取执行状态和阻塞项，这样能更快推进交付。

## 8. 验收标准

- 高频查询指令可以在已绑定身份前提下直接返回正确结果。
- 高风险动作必须先展示确认信息，再正式执行。
- 执行结果需要结构化输出，并标明下一步入口。
- 身份未绑定、对象不存在、权限不足、接口失败等场景均有明确兜底反馈。

## 9. 权限模型草案

- 模型负责识别意图，不负责最终授权。
- 服务端必须基于身份映射、角色和对象权限做最终校验。
- 禅道真实权限作为最终执行依据。

## 10. 高风险动作清单

- 发布状态更新
- 任务完成 / 关闭
- 需求关闭 / 评审结论写入
- 测试单状态更新
- 关闭准备 / 验收相关结论提交
