# 企微机器人/自建应用对话式操作禅道 PRD

## 1. 文档信息

- 产品名称：企微对话式禅道助手
- 文档版本：V1.0
- 目标：把现有 `openclaw-zentao-pack` 能力整理为可评审、可设计、可研发落地的产品需求文档

## 2. 背景

当前团队已经在云服务器 `1.14.73.166` 上沉淀了 `openclaw-zentao-pack`，包含禅道查询、创建、流转、企微回调、身份匹配、Prompt 治理和高频路由配置。现阶段需要把这些能力整理成一个对业务和研发都直观的产品方案，让“企微机器人/自建应用中通过对话操作禅道”具备明确的产品边界、交互形式和实施优先级。

## 3. 产品目标

### 3.1 核心目标

- 让用户在企业微信内直接完成禅道高频操作。
- 优先承接查询、轻量创建、状态流转和风险确认场景。
- 用统一对话体验承接机器人模式和自建应用模式。
- 所有敏感动作具备授权校验、二次确认和审计记录。

### 3.2 非目标

- 不在首期替代禅道 Web 后台。
- 不在首期覆盖全部复杂配置与后台管理动作。
- 不在首期支持多租户、多禅道环境大规模隔离。

## 4. 用户与角色

| 角色 | 使用目标 | 高频动作 |
| --- | --- | --- |
| 研发 | 快速查看和处理任务、Bug | 我的任务、开始任务、完成任务、看 Bug |
| 测试 | 快速提 Bug、看测试单和准出结果 | 创建 Bug、查测试单、测试准出 |
| 产品 | 基于 PRD 驱动产品、需求、迭代、任务落地 | 创建产品、导入 PRD、拆迭代、拆任务、评审需求 |
| 项目经理 | 快速查看执行、发布、验收状态 | 查执行、查发布、验收概览 |
| 负责人 | 审批或确认高风险操作 | 关闭 / 发布 / 状态流转确认 |

### 4.1 角色分工原则

- 机器人负责接收自然语言、PRD 文档、快捷命令，并产出摘要、草案和确认引导。
- 自建应用负责承接复杂表单、批量编辑、草案确认、审计查看和历史记录。
- 后端编排服务负责权限校验、幂等、审计，以及调用禅道 skill / API 正式写入数据。
- 所有会改动禅道正式数据的动作都必须经过“草案 -> 确认 -> 执行 -> 回执”的四段式闭环。

## 5. 目标场景

### 场景 A：碎片化查询

- 用户在聊天中快速输入：`我的任务`、`查执行 4`、`测试准出 4`
- 系统直接命中高频路由，快速返回结果卡片

### 场景 B：轻量创建

- 用户输入：`在执行 4 创建任务，标题是补齐企微映射`
- 系统补齐最少必要参数后直接创建

### 场景 C：风险流转

- 用户输入：`完成任务 33`
- 系统回显对象信息、状态变化和风险说明，要求确认

### 场景 D：复杂补录

- 用户输入：`提个 Bug`
- 如果需要填写较长复现步骤、附件、环境信息，则跳转自建应用表单页

### 场景 E：PRD 驱动交付规划

- 用户在机器人中发送产品需求文档 / PRD 链接 / 禅道需求文档 ID
- 机器人先解析文档，抽取产品、模块、需求点、验收点、风险项
- 机器人生成产品草案、迭代草案、研发任务草案，并给出摘要卡
- 用户在自建应用确认页中编辑负责人、工时、模块归属、迭代周期
- 用户确认后，由后端编排层调起禅道 skill / API 创建正式数据

## 7. 核心流程

### 7.1 查询流程

```text
用户发消息 -> 命中高频路由 -> 自动补齐上下文 -> 调用禅道查询 -> 返回列表 / 详情卡片
```

### 7.2 创建流程

```text
用户发消息 -> 识别创建对象 -> 缺字段则补问 -> 参数齐全后创建 -> 返回创建结果
```

### 7.3 风险确认流程

```text
用户发消息 -> 识别高风险动作 -> 校验权限 -> 输出确认卡 -> 用户确认 -> 执行 -> 返回回执并记录审计
```

### 7.4 PRD 到产品 / 任务创建流程

```text
用户发送 PRD / 需求文档
-> 机器人识别来源并抽取正文
-> 解析产品、模块、需求、验收点、风险
-> 生成产品草案、迭代草案、任务草案
-> 返回草案摘要卡
-> 用户点击“查看草案 / 去应用确认”
-> 自建应用打开确认页并允许编辑
-> 用户点击“确认创建并写入禅道”
-> 后端编排层调用禅道 skill / API 创建产品、需求、执行、任务
-> 机器人回执创建结果并给出下一步入口
```

### 7.5 草案状态机

| 状态 | 含义 | 可执行动作 |
| --- | --- | --- |
| `draft` | 已解析但未写入禅道 | 查看、编辑、重新拆分、校验 |
| `validated` | 已通过字段和权限校验 | 确认创建、回退编辑 |
| `committing` | 正在调用禅道创建 | 查看进度、禁止重复提交 |
| `committed` | 已成功写入禅道 | 查看结果、继续发起评审 / 分派任务 |
| `failed` | 创建失败或部分失败 | 查看失败原因、重试、人工修正 |

## 8. 角色工作流细化

### 8.1 产品经理

1. 发送 PRD / 产品需求文档。
2. 机器人解析并输出产品、模块、需求、迭代、任务摘要。
3. 用户点击查看草案，进入自建应用确认页。
4. 在确认页中编辑产品名、模块、负责人、工时、迭代计划、评审人。
5. 点击确认创建，系统写入禅道产品、需求、执行、任务。
6. 机器人返回创建结果，并提供“发起评审”“查看需求草稿”“查看任务分派”的下一步入口。

### 8.2 研发

1. 通过机器人查看我的任务和今天待办。
2. 在机器人中完成开始任务、完成任务、记录工时等高频动作。
3. 遇到复杂任务详情、自测补录、附件说明时跳自建应用详情页。
4. 任务完成后回到机器人确认状态流转，结果写入审计。

### 8.3 测试

1. 通过机器人查看测试范围、测试单和阻塞项。
2. 快速提 Bug 时先在机器人补轻量字段，再跳自建应用补复现步骤和附件。
3. 在机器人中查看准出摘要，在自建应用中查看完整报告与回归明细。

### 8.4 管理 / 项目经理

1. 通过机器人查看准备度、任务完成率、测试风险、发布检查结论。
2. 在自建应用驾驶舱中查看聚合统计、责任人、阻塞项和关闭条件。
3. 高风险发布 / 关闭动作统一经过确认卡与审计回执。

## 9. 页面 / 界面结构

### 9.1 机器人会话页

- 顶部：会话标题“禅道助手”
- 中部：消息流
- 消息类型：结果卡片、确认卡片、失败卡片、候选卡片
- 底部：输入框 + 快捷命令入口

### 9.2 自建应用工作台

- 工作台首页：我的任务、我的 Bug、待确认动作、快捷入口
- 详情页：对象信息 + 操作按钮 + 审计摘要
- 表单页：复杂创建 / 编辑场景承接

### 9.3 草案确认页

- 顶部：来源 PRD、当前草案版本、解析时间、状态标签
- 中部：产品信息、模块列表、需求列表、迭代规划、任务清单
- 交互：支持批量改负责人、批量设迭代、重新拆分、删除草案任务、新增人工任务
- 底部：`仅保存草案`、`重新解析`、`确认创建并写入禅道`

### 9.4 草案确认页关键规则

- 页面主键为 `draftId`，所有保存、校验、提交动作都必须带该标识。
- 任务表格需支持批量操作，至少支持批量改负责人、批量设迭代、批量设优先级。
- 用户人工改过的字段要打标，重新拆分时默认优先保留人工修改值。
- `确认创建并写入禅道` 前必须先通过校验接口，禁止前端只做本地假校验。
- 提交进入 `committing` 后页面应锁定，防止重复点击造成重复创建。

### 9.5 按角色展示的工作台权限规则

- 当用户切换到“按角色展示”模式时，工作台内容必须按角色权限裁剪，不能继续展示其他角色的功能入口。
- 产品角色仅保留：PRD 草案、需求草稿、评审发起、评审结论、验收概览、关闭建议。
- 研发角色仅保留：我的任务、我的 Bug、开始 / 完成任务、工时补录、自测回填。
- 测试角色仅保留：测试单、失败项、Bug 提报、回归验证、准出结论。
- 管理角色仅保留：项目准备度、任务完成率、测试风险、上线条件、关闭清单。
- 跨角色视图仅保留：交接物、责任链、上线闭环，不提供单角色对象录入入口。
